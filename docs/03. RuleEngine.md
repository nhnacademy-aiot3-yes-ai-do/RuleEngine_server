# 03. RuleEngine

CollectorService로부터 전달받은 센서 데이터를, 해당 재배 환경에 설정된 임계값과 비교하여 정상 범위인지 이상 상황인지 판단합니다.

## 센서/임계값 정보 동기화

센서 등록 정보 및 재배 환경별 임계값은 다른 서비스에서 관리하며, 해당 정보가 등록·변경될 때마다 RabbitMQ로 발행됩니다. 이를 `@RabbitListener`로 구독하여 우리 쪽 Redis에 반영해두고, CollectorService의 검증(`SensorDataValidator`)과 RuleEngine의 판단 시 이 Redis를 조회하여 사용합니다.

```
[다른 서비스: 센서 등록 / 임계값 변경]
        ↓
[RabbitMQ 발행]
        ↓
[@RabbitListener 수신]
        ↓
[우리 쪽 Redis에 반영]

──────────────────────────────

[CollectorService 검증 / RuleEngine 판단 시]
        ↓
[Redis 조회하여 사용]
```

- 센서 등록 정보와 임계값 정보의 원천(source of truth)은 다른 서비스에 있으며, 우리 쪽 Redis는 그 정보를 조회 용도로 동기화해둔 것입니다.
- 아래 A(DB/Redis 조회), C(이벤트 방식)에서 조회하는 Redis 또는 인메모리 상태는 모두 이 동기화 과정을 거쳐 반영된 데이터를 기반으로 합니다.

## 전체 흐름

`SensorDataReadyEvent`로 수신하는 데이터는 이미 파싱·정제·검증이 완료된 상태입니다. 이 데이터를 해당 재배 환경에 설정된 임계값과 비교하여 정상/이상 여부를 판단한 뒤, 판단 결과에 따라 다음과 같이 처리합니다.

```
[SensorDataReadyEvent 수신 (파싱·정제·검증 완료)]
        ↓
[해당 재배 환경의 임계값과 비교]
        ↓
        ├─→ [모든 데이터] → RabbitMQ 발행 (실시간 데이터용)
        │
        ├─→ [임계값을 벗어난 데이터] → RabbitMQ 발행 (알림용)
        │
        └─→ [임계값을 벗어난 데이터] → 제어 명령 발행 (센서 타입별, ex) 습도가 낮으면 가습 명령)
```

- 모든 데이터는 실시간 데이터용 큐로 발행되고, 임계값을 벗어난 데이터는 이와 별도로 알림용 큐로 한 번 더 발행됩니다. 즉 임계값을 벗어난 데이터는 두 큐 모두에 발행되는 것이 의도된 동작입니다.
- 임계값을 벗어난 경우, 알림 발행과 별개로 해당 센서 타입에 대응되는 제어 명령을 발행합니다. (ex) 습도가 낮으면 가습, 온도가 높으면 환기 등) 제어 명령의 전달 방식(RabbitMQ 등)은 아직 확정되지 않았습니다.

## 구조

위 흐름에서 "임계값과 비교하여 판단"하는 부분을 어떻게 구현할지에 대해, 재배 환경 종류나 개수가 늘어나더라도 공통 로직을 유지할 수 있는 방식을 고려하여 아래 세 가지를 검토 중입니다.

### A. DB(Redis) 조회 방식

판단 로직은 하나로 고정하고, 센서 데이터가 들어올 때마다 재배 환경 ID를 기준으로 Redis에 저장된 임계값을 조회하여 비교합니다.

```
[SensorDataReadyEvent 수신]
        ↓
[재배 환경 ID 확인]
        ↓
[Redis에서 해당 재배 환경의 임계값 조회]
        ↓
[조회한 임계값과 센서 데이터 비교]
        ↓
[정상 / 이상 판단 → 위 전체 흐름에 따라 후속 처리]
```

- 판단 시점마다 조회가 발생하므로 항상 최신 임계값을 기준으로 판단할 수 있습니다.
- 재배 환경이 몇 개든 노드/코드 구조가 늘어나지 않습니다.

### B. 노드 방식

재배 환경이 등록될 때마다, 해당 환경 전용 판단 노드를 동적으로 생성하여 그래프에 연결합니다. 이후 그 재배 환경의 센서 데이터는 생성된 전용 노드로 흘러가 판단이 이루어집니다.

```
[재배 환경 등록됨]
        ↓
[해당 재배 환경 전용 판단 노드 생성 및 그래프에 연결]

──────────────────────────────

[SensorDataReadyEvent 수신]
        ↓
[재배 환경 ID를 기준으로 대응되는 판단 노드로 라우팅]
        ↓
[해당 노드 내에 미리 반영된 임계값으로 비교]
        ↓
[정상 / 이상 판단 → 위 전체 흐름에 따라 후속 처리]
```

- 재배 환경 수만큼 노드가 늘어나므로, 노드 생성·삭제 및 서버 재시작 시 재구성 로직이 필요합니다.
- 노드 자체가 재배 환경 단위로 명시적으로 존재하여, FBP 구조를 더 직접적으로 드러낼 수 있습니다.

### C. 이벤트 방식

B(노드 방식)와 마찬가지로 재배 환경마다 판단 노드가 존재하되, 노드 간의 연결을 직접적인 메서드 호출 체인이 아니라 이벤트 발행/구독으로 구성합니다. 03번 문서(CollectorService → RuleEngine 전달)에서 사용한 것과 동일한 방식을, RuleEngine 내부의 노드 간 연결에도 적용하는 방식입니다.

```
[재배 환경 등록됨]
        ↓
[해당 재배 환경 전용 판단 리스너 등록]

──────────────────────────────

[SensorDataReadyEvent 수신]
        ↓
[재배 환경 ID에 대응되는 이벤트 발행]
        ↓
[해당 재배 환경 전용 리스너가 이벤트 수신]
        ↓
[리스너 내에 미리 반영된 임계값으로 비교]
        ↓
[정상 / 이상 판단 → 위 전체 흐름에 따라 후속 처리]
```

- B(노드 방식)와 마찬가지로 재배 환경 수만큼 리스너가 늘어나므로, 등록·삭제 및 서버 재시작 시 재구성 로직이 필요합니다.
- 노드 간 연결을 이벤트로 느슨하게 구성하여, 각 판단 리스너가 서로의 존재를 몰라도 되는 구조를 가질 수 있습니다.

## 검토 방향

세 가지 방식 중 하나를 채택하거나, 각 방식의 장단점 및 성능을 비교하여 정리하는 것도 함께 고려하고 있습니다.
