# 00. 아키텍처 개요

버섯 재배 환경의 센서(온도, 습도, CO2 등)가 MQTT로 데이터를 발행하면, CollectorService가 이를 구독하여 파싱·정제한 뒤 이벤트로 RuleEngine에 전달합니다. RuleEngine은 전달받은 데이터를 각 재배 환경에 설정된 임계값과 비교하여 정상/이상 여부를 판단하고, 그 결과를 RabbitMQ로 발행합니다.

## 구간별 설명

### 1. CollectorService (MQTT 구독 및 파싱·정제)

센서로부터 발행되는 데이터를 MQTT로 구독하고, 수신한 원본 데이터를 파싱·정제하여 이후 처리에 사용할 수 있는 형태로 변환합니다.

- 참고 : [01. CollectorService 구조](./01.%20CollectorService%20구조.md)

### 2. CollectorService → RuleEngine 데이터 전달

CollectorService와 RuleEngine은 하나의 서버로 두기로 결정하였으며, `@Async` + `@EventListener` 기반으로 데이터를 전달합니다.

- 참고 : [02. 데이터 전달(EventListener)](./02.%20데이터%20전달(EventListener).md) 참고.

### 3. RuleEngine (판단)

전달받은 센서 데이터를, 해당 재배 환경에 설정된 임계값과 비교하여 정상 범위인지 이상 상황인지 판단합니다.

임계값을 관리하는 방식으로 다음 세 가지를 검토 중입니다.

- **DB 조회 방식**: 판단 로직은 하나로 고정하고, 재배 환경 ID를 기준으로 임계값을 DB에서 조회하여 비교
- **노드 방식**: 재배 환경이 등록될 때마다 해당 환경 전용 필터 노드를 동적으로 생성하여 그래프에 연결
- **이벤트 방식**: 재배 환경 등록/수정 시 임계값 정보를 이벤트로 발행하고, 이를 수신하여 인메모리 상태로 유지하며 판단 시 조회

세 가지 중 하나를 채택하거나, 각 방식의 장단점 및 성능을 비교하여 정리하는 것도 함께 고려하고 있습니다.

- 참고 : [03. RuleEngine 구조](./03.%20RuleEngine.md)

### 4. RuleEngine → RabbitMQ (발행)

RuleEngine의 판단 결과를 RabbitMQ로 발행하여, 이후 이 데이터를 필요로 하는 서비스(알림, 저장 등)가 소비할 수 있도록 합니다.

- 참고 : [04. RabbitMQ 발행](./04.%20RabbitMQ%20발행.md) 참고.
