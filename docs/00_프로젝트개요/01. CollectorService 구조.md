# 01. CollectorService 구조

센서로부터 발행되는 데이터를 MQTT로 구독하고, 수신한 원본 데이터를 파싱·정제하여 이후 처리에 사용할 수 있는 형태로 변환합니다.

### A. MQTT

모든 센서가 아래 구조로 데이터를 발행한다고 가정합니다.

```
{cultivation_id}/{device_eui}/{value_type} {"value":...,"time":...}
```

발행된 메세지를 MQTT 구독을 통해 수신하여 가져옵니다. 수신한 토픽과 페이로드는 `CollectorService.ingest(topic, payload)`로 전달됩니다.

상황에 따라 다음을 고려합니다.
- 발행 시 토픽 구조 수정 (ex) 센서 데이터 발행 시, 토픽은 항상 `mushroom`으로 시작)
    - 이와 연계되어 발행 토픽 수정에 따른 구독 토픽 변경 또한 고려합니다. (ex) `mushroom/#`와 같이 와일드카드를 사용하여, 토픽의 최상단 계층이 일치하는 메세지에 대해서만 수신)

### B. 파싱

수신한 데이터에 대해 파싱을 수행하여 DTO로 전환합니다. 토픽 문자열에서 `cultivation_id`, `device_eui`, `value_type`을 추출하고, 페이로드(JSON)에서 `value`, `time`을 추출하여 하나의 DTO로 합칩니다.

```java
import com.fasterxml.jackson.annotation.JsonFormat;
import lombok.AllArgsConstructor;
import lombok.Getter;
import lombok.NoArgsConstructor;
import lombok.Setter;

import java.time.LocalDateTime;

@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor

    private ValueType valueType;

```

public enum ValueType {

### C. 정제 및 검증

수신한 메세지를 DTO로 전환한 뒤, 정제 및 검증을 진행합니다.
- 해당 `device_eui`가 실제로 등록된 센서인지 확인
- 등록된 센서 정보의 `value_type`, 센서명 등이 수신한 데이터와 일치하는지 확인
- `value`가 해당 센서 타입의 물리적으로 유효한 범위 내의 값인지 확인 (ex) 온도가 -10 ~ 50도 범위를 벗어나는 등 명백히 오류인 값)
- 검증에 실패한 데이터는 다음 단계로 전달하지 않고, 로그를 남긴 뒤 폐기

검증에 필요한 센서 등록 정보(센서명, 타입, 위치 등)는 Redis에 저장되어 있으며, 이를 조회하여 비교합니다. `CollectorService`가 이 컴포넌트를 주입받아 호출합니다.
