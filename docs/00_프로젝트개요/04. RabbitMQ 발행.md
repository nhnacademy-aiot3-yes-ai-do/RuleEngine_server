# 04. RabbitMQ 발행

- 참고 : [RabbitMQ](https://github.com/ysh011130/RabbitMQ)

RuleEngine의 판단 결과를 RabbitMQ로 발행하여, 이후 이 데이터를 필요로 하는 서비스(실시간 데이터 표시, 알림, 장치 제어 등)가 소비할 수 있도록 합니다.

## 발행 대상

판단 결과에 따라 다음 세 가지 용도로 발행이 이루어집니다.

- **실시간 데이터용**: 정상/이상 여부와 무관하게 모든 센서 데이터
- **알림용**: 임계값을 벗어난 데이터
- **제어 명령용**: 임계값을 벗어난 경우, 이를 해소하기 위한 센서 타입별 명령 (ex) 습도가 낮으면 가습 명령)

임계값을 벗어난 데이터는 실시간 데이터용과 알림용 양쪽에 모두 발행되는 것이 의도된 동작입니다.

## 큐 구성

| 용도 | 큐 개수 |
|---|------|
| 실시간 데이터 | 1개 |
| 알림 | 1개 |
| 제어 명령 | 미정 |

### A. 실시간 데이터용 큐

정상/이상 여부와 무관하게 모든 데이터가 여기로 발행됩니다. 소비 측(대시보드 등)이 데이터를 구분해서 받을 필요가 없으므로, 하나의 큐로 통합합니다.

### B. 알림용 큐

임계값을 벗어난 데이터만 발행됩니다.

**단일 큐로 통합**: 이상 종류(온도/습도/CO2 등)와 무관하게 하나의 알림 큐로 발행하고, 소비 측에서 메시지 내용을 보고 어떤 이상인지 구분하여 처리

- 고려해볼 수 있는 사항
  - **이상 종류별로 큐 분리**: 온도 이상, 습도 이상 등으로 큐를 나누어 발행하고, 소비 측이 필요한 큐만 선택적으로 구독 
  - 알림을 실제로 소비하는 쪽(이메일, 실시간 알림 등)이 이상 종류에 따라 처리 방식이나 채널을 달리해야 하는지에 따라 결정이 달라질 수 있습니다.

### C. 제어 명령용 큐

임계값을 벗어난 경우, 이를 해소하기 위한 명령을 발행합니다. 센서 타입별로 명령의 내용이 다르므로 아래와 같은 경우의 수를 검토 중입니다.

- **단일 `command` 큐로 통합**: 모든 제어 명령을 하나의 큐로 발행하고, 메시지 안에 "무엇을, 어떻게 하라"는 정보(센서 타입, 명령 종류 등)를 담아 소비 측에서 이를 확인하여 분기 처리
- **센서 타입별로 큐 분리**: 온도 제어용, 습도 제어용 등으로 큐를 나누어 발행하고, 각 큐를 해당 장치 제어 서비스가 개별적으로 구독

Exchange를 하나로 두고 routing key로 구분하는 방식(ex) `command.humidity`, `command.temperature`)을 사용하면, 위 두 경우를 같은 Exchange 구조 위에서 구현할 수 있습니다. 다만 제어 명령 전달에 RabbitMQ를 사용할지 여부 자체가 아직 확정되지 않았습니다.
